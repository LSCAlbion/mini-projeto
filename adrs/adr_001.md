# 1 ADR: Escolha do Banco de Dados Principal para o Motor de Recomendação do EduVerse

## 2 Status
- [ ] Proposto
- [x] Aceito
- [ ] Depreciado
- [ ] Rejeitado

## 3 Contexto
O desafio técnico atual é definir como armazenar e recuperar milhares de logs de interação dos alunos e metadados heterogêneos de conteúdo (vídeos, quizzes, textos) em tempo real. A decisão é crítica agora para viabilizar a infraestrutura inicial e garantir que a IA consiga acessar esses dados para gerar recomendações personalizadas com latência inferior a 200ms (conforme o RNF01) sob alta escalabilidade.

## 4 Decisão
Decidimos adotar um banco de dados NoSQL Orientado a Documentos (MongoDB) para o subsistema de recomendação e catálogo de conteúdos, porque ele oferece flexibilidade de *schema* nativa e alta taxa de transferência para leitura/escrita de grandes volumes de dados não-estruturados.

## 5 Consequências (Trade-offs)
**Ganhos (+):**
* (+) **Performance:** Consultas de metadados e histórico de usuário muito mais rápidas, sem a necessidade de múltiplos `JOINs`, atendendo ao RNF de latência da IA.
* (+) **Escalabilidade:** Facilidade de particionamento horizontal (*sharding*) nativo, permitindo suportar os picos de milhares de alunos simultâneos sem degradação do serviço.

**Perdas (-):**
* (-) **Confiabilidade (Consistência):** Sacrificamos as garantias estritas de transações ACID (típicas de bancos relacionais). O gerenciamento da integridade de relacionamentos complexos terá que ser tratado via código na camada de aplicação (*eventual consistency*).

## 6 Alternativas Consideradas
**Alternativa:** PostgreSQL (SQL tradicional).

**Rejeitado porque:** O modelo relacional rígido exigiria normalização excessiva e migrações constantes para cada novo formato de conteúdo educacional inserido no sistema. Isso geraria um grande gargalo de performance devido à quantidade de `JOINs` necessários para montar o perfil do aluno antes de enviá-lo para a API de Inteligência Artificial.
